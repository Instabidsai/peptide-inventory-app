# Full Application Audit — ThePeptideAI
**Date:** 2026-03-03
**Scope:** Full-stack audit across security, frontend, backend, database, testing, performance, and configuration

## Executive Summary

The application is a **well-architected multi-tenant SaaS platform** with solid foundations: proper code splitting, consistent auth patterns, comprehensive RLS policies (224 across 55 tables), and modern tooling. However, the audit uncovered **3 critical security issues**, **12 high-severity items**, and numerous medium/low improvements across all layers.

## 1. CRITICAL Security Findings (Fix Immediately)

### 1.1 Git History Contains Committed Secrets
**Severity: CRITICAL** | **Status: Known (documented in CLAUDE.md)**

Secrets were previously committed to git. Anyone with repo access can extract them from history.

**Action:**
```bash
bfg --replace-text secrets.txt
git reflog expire --expire=now --all && git gc --prune=now --aggressive
git push origin --force --all
```

### 1.2 `dangerouslySetInnerHTML` with Unsafe DOMPurify Config
**Severity: CRITICAL** | **Files:** `src/components/protocol-builder/RichProtocolPreview.tsx:27-30`, `src/pages/client/ClientResources.tsx`

```typescript
// CURRENT — allows CSS injection
DOMPurify.sanitize(html, { ADD_TAGS: ['style'], ADD_ATTR: ['target', 'rel'] });
```

`ADD_TAGS: ['style']` allows attackers to inject CSS that can overlay fake login forms, hide content, or harvest clicks.

**Fix:** Remove `ADD_TAGS: ['style']` or use a strict allowlist:
```typescript
DOMPurify.sanitize(html, {
  ALLOWED_TAGS: ['b','i','em','strong','p','br','ul','ol','li','a','h1','h2','h3','table','tr','td','th'],
  ALLOWED_ATTR: ['href','target','rel']
});
```

### 1.3 Session Tokens in localStorage (XSS → Session Hijacking)
**Severity: CRITICAL** | **File:** `src/integrations/sb_client/client.ts:19`

Auth tokens are stored in `localStorage` and `sessionStorage`. Any XSS vulnerability gives full account takeover.

**Action:** Migrate to httpOnly cookies (Supabase supports this) and implement strict Content Security Policy headers.

## 2. HIGH Severity Findings

### 2.1 No Rate Limiting on Vercel API Routes
**Files:** All files in `api/`

`/api/checkout/create-session.ts`, `/api/shipping/create-label.ts`, `/api/email/send.ts` have zero rate limiting. This enables:
- Shipping label fraud
- Email spam
- Payment session flooding

**Fix:** Add rate limiting middleware (Upstash/Redis-based for serverless, or Supabase table-based).

### 2.2 OAuth Tokens in sessionStorage
**File:** `src/contexts/AuthContext.tsx:170-180`

WooCommerce/Shopify/Composio OAuth tokens are stored client-side. Any XSS steals integration credentials.

**Fix:** Store OAuth tokens server-side in `tenant_api_keys` with encryption.

### 2.3 In-Memory Rate Limiter Resets on Cold Start
**File:** `supabase/functions/_shared/rate-limit.ts:11`

Edge function rate limiting uses an in-memory `Map`. Every cold start (deploy, idle timeout) resets all limits.

**Fix:** Move rate limit tracking to a Supabase table or use Deno KV.

### 2.4 Webhook Secret Not Per-Tenant
**Files:** `api/webhooks/stripe.ts`, `api/webhooks/psifi.ts`

A single webhook signing secret is used for all tenants. If compromised, all tenants are affected.

**Fix:** Store tenant-specific webhook secrets in `tenant_api_keys`.

### 2.5 Stripe Webhook Silent Failures
**File:** `api/webhooks/stripe.ts:114-147`

If the DB upsert for `tenant_subscriptions` fails, the webhook returns 200 (success). Stripe won't retry. The subscription is silently never created.

**Fix:** Return 500 on DB errors so Stripe retries.

### 2.6 Unvalidated External API Response Parsing
**File:** `supabase/functions/check-payment-emails/index.ts:434-447`

Composio email responses are parsed through multiple fallback paths with no shape validation. Malformed data could cause unexpected behavior.

**Fix:** Add a type guard function to validate each email object before processing.

### 2.7 Hardcoded Shipping Address Fallback Leaks Cross-Tenant
**File:** `api/shipping/get-rates.ts:44-51`

If a tenant hasn't configured `ship_from_*` in `tenant_config`, it falls back to environment variables (which are PureUSPeptide's address). Orders could ship from the wrong address.

**Fix:** Require tenant_config; return 400 if not configured.

### 2.8 PsiFi JSON Parsing Unguarded
**File:** `api/checkout/create-session.ts:139`

If PsiFi returns non-JSON, `productRes.json()` throws and is uncaught.

**Fix:** Wrap in try/catch, validate `product.id` exists.

### 2.9 Missing Indexes on Commissions Table
**Impact:** Every commission query does a full table scan.

```sql
-- Add these immediately
CREATE INDEX idx_commissions_org_id ON commissions(org_id);
CREATE INDEX idx_commissions_status_created ON commissions(status, created_at DESC);
CREATE INDEX idx_commissions_partner_org ON commissions(partner_id, org_id);
```

### 2.10 Missing Indexes on Notifications Table
**Impact:** Every user session queries unread notifications without an index.

```sql
CREATE INDEX idx_notifications_user_read ON notifications(user_id, is_read);
CREATE INDEX idx_notifications_user_created ON notifications(user_id, created_at DESC);
```

### 2.11 Test Coverage Threshold at 15%
**File:** `vitest.config.ts`

The coverage threshold is 15%. Critical business logic (commissions, fulfillment, multi-level referrals) has no tests.

**Fix:** Raise threshold to 50% and add tests for commission calculations, order fulfillment, and RLS policy enforcement.

### 2.12 Two Tables Missing RLS Entirely
**Tables:** `newsletter_subscribers`, `supplements`

These tables have no Row Level Security policies. Any authenticated user can read/write.

**Fix:** Enable RLS and add appropriate policies.

## 3. MEDIUM Severity Findings

### 3.1 TypeScript Base Config Too Loose
**File:** `tsconfig.json`

```json
"noImplicitAny": false,
"strictNullChecks": false,
"noUnusedLocals": false,
"noUnusedParameters": false
```

The app tsconfig is strict, but the base tsconfig is loose, creating inconsistency. Root-level type checks miss errors.

**Fix:** Enable strict options in the base config.

### 3.2 Duplicate OrderDetails Files (3,230 LOC wasted)
**Files:** `src/pages/sales/OrderDetails.tsx` (1,644 LOC) and `OrderDetailsV2.tsx` (1,586 LOC)

These are 95% identical. OrderDetailsV2 has a cleaner pattern with extracted sub-components.

**Fix:** Remove `OrderDetails.tsx`, keep `OrderDetailsV2.tsx`.

### 3.3 Oversized Components (10+ files >1,000 LOC)

| File | Lines | Recommendation |
|------|-------|----------------|
| `admin/PartnerDetail.tsx` | 1,812 | Extract ReferralCard, CommissionCard, DownlineCard |
| `sales/OrderDetails.tsx` | 1,644 | Remove (duplicate of V2) |
| `sales/OrderDetailsV2.tsx` | 1,586 | Extract OrderHeader, OrderItems, Shipping, Payment |
| `admin/Reps.tsx` | 1,535 | Extract TableSection, FilterSection, ModalSection |
| `admin/SystemHealth.tsx` | 1,504 | Extract per-category health cards |
| `Integrations.tsx` | 1,373 | Extract per-integration sections |
| `admin/AdminDashboard.tsx` | 1,217 | Extract dashboard cards |
| `hooks/use-sales-orders.ts` | 1,199 | Acceptable for complex hook |
| `Settings.tsx` | 1,128 | Extract setting section components |
| `Peptides.tsx` | 1,054 | Extract catalog and wholesale views |
| `components/SimpleVials.tsx` | 1,008 | Extract vial state logic to useMemo |

### 3.4 Unsafe Error Catch Blocks (10+ instances)
**Pattern found across:** `Settings.tsx`, `Bottles.tsx`, `Integrations.tsx`, `PartnerDetail.tsx`

```typescript
// Current — unsafe
catch (err) { toast({ description: (err as any)?.message }); }

// Fix
catch (err) { toast({ description: err instanceof Error ? err.message : 'Unknown error' }); }
```

### 3.5 Empty Catch Blocks Swallow Errors
**Files:** `Lots.tsx:132,140`, `Finance.tsx:147`, `FeatureManagement.tsx:62`, `Reps.tsx:624,844,1105`

```typescript
catch { /* onError in hook shows toast */ }
```

These hide unexpected errors. At minimum, add `console.error`.

### 3.6 Missing React.memo on Expensive Components
Only 1 file uses `React.memo` in the entire codebase. Large card components inside PartnerDetail, OrderDetails, and AdminDashboard re-render unnecessarily.

**Fix:** Add `React.memo` to pure card/section components.

### 3.7 N+1 Query in PartnerDetail.tsx
**Lines 47-79:** Two separate queries for clients and orders, then a JavaScript O(n^2) loop to aggregate.

**Fix:** Replace with a single RPC function or JOIN query.

### 3.8 Missing CSRF Protection
No CSRF tokens on state-changing forms. Combined with localStorage token storage, this increases attack surface.

### 3.9 Overly Permissive RLS Policies
```sql
-- resource_themes: ANY authenticated user can manage themes
CREATE POLICY "Authenticated users can manage themes" ON resource_themes
  FOR ALL USING (auth.role() = 'authenticated');
```

Similar issue on `resources.UPDATE`. Should be org-scoped or role-restricted.

### 3.10 No Rollback (DOWN) Migrations
127 migration files with no rollback procedures. If a deployment fails, manual SQL is required.

### 3.11 Missing Foreign Key Constraints
- `commissions.sale_id` — no FK to `sales_orders(id)`
- `commissions.partner_id` — no FK to `profiles(id)`
- `client_daily_logs.contact_id` — no FK to `contacts(id)`
- `protocol_logs.protocol_item_id` — no FK to `protocol_items(id)`

### 3.12 Hardcoded Supabase URL in Vite Config
**File:** `vite.config.ts`

```typescript
'/functions/v1': { target: 'https://mckkegmkpqdicudnfhor.supabase.co' }
```

Should use `process.env.VITE_SUPABASE_URL`.

### 3.13 Sentinel Config Constants Hardcoded
**File:** `supabase/functions/sentinel-worker/index.ts:30-40`

All thresholds (`CIRCUIT_BREAKER_THRESHOLD`, `MAX_BUGS_PER_RUN`, etc.) are hardcoded. Changing them requires redeployment.

**Fix:** Store in a `sentinel_config` table.

### 3.14 Health-Probe Doesn't Test RLS
**File:** `supabase/functions/health-probe/index.ts`

Uses service role key, so RLS is bypassed. Database could be "healthy" while users can't access data.

**Fix:** Add an RLS test using a test user JWT.

### 3.15 Card Fee Rate Hardcoded at 3%
**File:** `api/checkout/create-session.ts:102`

Should come from `tenant_config` so tenants can configure their fee pass-through.

### 3.16 Stripe Price IDs Not Seeded
Each subscription plan needs real Stripe price IDs. Currently empty, blocking subscription creation.

### 3.17 WooCommerce Webhook Variable Shadowing + Missing Error Handling
**File:** `api/webhooks/woocommerce.ts:100,118`

Inner `data` variable shadows outer. No error handling on contact update/insert.

### 3.18 Error Messages Leak Internal Details
**File:** `api/checkout/create-session.ts:136`

```typescript
return res.status(502).json({ error: 'Payment processor error', psifi_error: errBody });
```

`psifi_error` exposes payment processor internals to the client.

## 4. LOW Severity Findings

### 4.1 No Prettier/Formatter Configured
Only ESLint runs on pre-commit. No code formatting enforcement.

**Fix:** Add Prettier + `format` / `format:check` scripts.

### 4.2 Missing ESLint Accessibility Rules
No `eslint-plugin-jsx-a11y` configured. Accessibility issues won't be caught during development.

### 4.3 Docker Agent-API Runs as Root
**File:** `agent-api/Dockerfile`

No `USER` directive. Container runs as root.

**Fix:** Add `RUN useradd -m appuser && USER appuser`.

### 4.4 No Web Vitals / Performance Monitoring
No LCP, FID, CLS tracking. No performance budget alerts.

### 4.5 No Image Optimization Pipeline
All images served as PNG/JPG. No WebP conversion, no responsive sizes, no image CDN.

### 4.6 `legacy-peer-deps=true` in .npmrc
Bypasses peer dependency validation. May hide real compatibility issues.

### 4.7 `preflight` Script Doesn't Run Tests
```json
"preflight": "npm run typecheck && npm run lint && npm run cycles && npm run build"
```

Should include `npm run test` before build.

### 4.8 No Unique Constraints on Natural Keys
`contacts.email` and `peptides.sku` should be unique per org.

### 4.9 Missing Indexes on Health/Nutrition Tables
`body_composition_logs`, `meal_logs`, `protocol_items`, `protocol_logs`, `expenses` — all missing indexes for common query patterns.

### 4.10 CORS Origin Fallback Leaks Information
**File:** `supabase/functions/_shared/cors.ts`

When origin isn't in the whitelist, it falls back to the first allowed origin rather than returning no CORS header.

## 6. Architecture Strengths (Keep Doing These)

- **Multi-tenancy isolation** — org_id on all 57 tables with RLS enforcement
- **Code splitting** — All 50+ routes use lazy loading with retry logic
- **Auth pattern** — 3-layer auth (header → JWT → role) consistently applied
- **Webhook signature verification** — Timing-safe comparison used correctly
- **TanStack Query config** — Sensible defaults (1min stale, no refetch on focus)
- **Self-healing system** — Comprehensive autonomous error detection and repair
- **Edge function auth** — `verify_jwt = false` with manual auth avoids race conditions
- **Vercel security headers** — HSTS, X-Frame-Options, CSP all configured
- **No console.log in production** — Clean production output
- **Form validation** — Consistent react-hook-form + Zod pattern throughout
